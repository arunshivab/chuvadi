# Chuvadi.Pdf.Fuzz

A random-input mutation fuzzer for Chuvadi's parser surfaces.

## What this does (and doesn't)

This is a **random-input fuzzer**, not a coverage-guided one. Each iteration:

1. Picks a seed from the on-disk corpus
2. Applies one or more random mutations (bit flips, byte replacements, insertions,
   deletions, "interesting value" substitutions, truncations, range duplications,
   splices with other seeds)
3. Feeds the mutated bytes to the target
4. If the target throws one of its documented exception types → **expected**
   (cleanly-rejected input, this is fine)
5. If the target throws anything else (NullReferenceException, IndexOutOfRangeException,
   ArgumentOutOfRangeException, OverflowException, etc.) → **crash**, input saved to
   `crashes/<target>/<sha256>.bin` for triage

This catches *shallow* bugs efficiently — null derefs, OOB reads, infinite loops via
timeout, unchecked arithmetic, and parsing crashes on truncated input. It does NOT do
coverage feedback (libFuzzer-style), so deep state-machine bugs that require many
correct prior steps to reach won't surface as readily. For coverage-guided fuzzing,
SharpFuzz (NuGet) wrapping libFuzzer is the right tool; Chuvadi avoids that NuGet
dependency.

## Targets

| Name             | What it fuzzes                                         |
|------------------|--------------------------------------------------------|
| `pdf-open`       | `PdfDocument.Open(Stream)` + page enumeration          |
| `content-stream` | `ContentStreamParser.Parse(byte[])` with stub resolver |
| `truetype`       | `new TrueTypeLoader(byte[])` + a couple of lookups     |

Discover them via `dotnet run -c Release -- --list`.

## Running locally

```bash
cd tests/Chuvadi.Pdf.Fuzz

# Fuzz the PDF parser for 60 seconds
dotnet run -c Release -- pdf-open --duration 60

# Fuzz the content stream parser for 5 minutes
dotnet run -c Release -- content-stream --duration 300

# Reproducible run (fixed seed)
dotnet run -c Release -- truetype --duration 60 --seed 42

# Larger inputs (default 64 KiB)
dotnet run -c Release -- pdf-open --duration 60 --max-input-size 1048576
```

## Reading a crash report

Each crash produces two files in `crashes/<target>/`:

- `<sha256>.bin` — the exact input bytes that triggered the crash
- `<sha256>.txt` — exception type, message, full stack trace

The SHA-256 is over the input bytes, so identical crashes from different runs collide
to the same file — useful for deduplication.

To reproduce a crash deterministically, copy the `.bin` into a unit test that loads the
file and re-runs the target. Example skeleton:

```csharp
[Fact]
public void Regression_pdf_open_<hash>_should_not_crash()
{
    byte[] input = File.ReadAllBytes("crashes/pdf-open/<hash>.bin");
    using MemoryStream stream = new(input);
    Action act = () => PdfDocument.Open(stream);
    // After the fix:
    act.Should().NotThrow<NullReferenceException>();
    // The target may still legitimately reject the input — we just don't want a crash:
    act.Should().Throw<PdfDocumentException>();
}
```

## Regenerating the seed corpus

The PDF corpus is synthesized from `PdfDocumentBuilder`. Regenerate when the writer's
output format changes meaningfully:

```bash
dotnet run -c Release -- --regenerate-corpus
```

The content-stream corpus is also regenerated by this command (ASCII byte sequences).
The TrueType corpus is **not** pre-populated. Chuvadi doesn't have a TTF writer, and
the test repo doesn't currently include sample TTF binaries. The fuzzer will start with
its zero-byte default and mutate from there — perfectly capable of finding shallow header
parsing bugs but unable to reach deep table-parsing code without a valid seed. If you have
a permissively-licensed TTF you want to use as a seed, drop it into
`corpus/truetype/seed.bin`.

## CI

A scheduled GitHub Actions workflow (`.github/workflows/fuzz.yml`) runs all three
targets weekly (Sunday 02:00 UTC) for 15 minutes each. Any crashes found are uploaded
as workflow artifacts and the job fails.

The workflow is **not** required for PR merge — fuzzing is too slow to gate normal
development on.

## When the fuzzer finds something

1. Triage the crash file. Confirm reproducibility by re-running the target with that
   single input (TODO: add a `--repro <file>` mode).
2. Write a regression test in the appropriate test project, loading the crash `.bin`.
3. Fix the bug.
4. Verify the regression test passes; verify the fuzzer no longer reproduces the crash.
5. Move the crash file to `tests/<module>.Tests/crashes/` so it's preserved as a
   permanent regression seed.

## Limitations to be honest about

- No coverage feedback. Deep bugs requiring sequential correct steps are missed.
- No structure-aware mutation. Random bytes rarely produce valid PDF/TTF structures.
  The mutation engine ameliorates this with corpus splicing (combining valid prefixes
  with mutated suffixes), but it's not as effective as grammar-based fuzzing.
- The expected-exception list is permissive. If the parser throws e.g.
  `EndOfStreamException` from somewhere it shouldn't, the fuzzer won't flag it.
  Periodically auditing the expected-exception list is worth doing.
- The runner is single-threaded. Real coverage-guided fuzzers run many workers in
  parallel.

These limitations are why ongoing CI runs are valuable: they accumulate iteration count
over weeks/months, eventually surfacing bugs that a single short run misses.
