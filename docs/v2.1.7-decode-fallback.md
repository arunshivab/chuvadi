# v2.1.7 — DisplayListBuilder.DecodeText silently falls through to Latin-1 byte-passthrough when font resolution fails

## Severity

High. Affects any PDF whose font dictionary cannot be resolved through the
`DisplayListBuilder` lookup chain. The failure is **silent** — no exception,
no log — and produces text that looks structurally correct (right number of
glyphs, right positions) but with character content equal to the raw byte
codes from the content stream rather than their decoded Unicode.

Visible to users only when low byte codes (typical of Ghostscript-subset
simple fonts using a custom `/Differences` encoding starting at code 1) hit
the SVG output and most get replaced by spaces by the XML writer (codes
outside `\u0009 \u000A \u000D \u0020-\uFFFD` are invalid in XML 1.0). For
high byte codes (most ordinary PDFs whose subsets start at code 0x20+) the
bug is currently invisible because Latin-1 passthrough *coincides* with the
correct Unicode for codes 0x20-0x7E — which means we don't know how many
real-world PDFs are silently mis-decoding.

## Reproducer

Ghostscript-produced Type1C PDF from a LibreOffice source:

1. Source: `cff_writer.odt` — one line of text "The quick brown fox 0123456789"
   in font Frank Ruhl Hofshi (any OTF/.otf font with a custom subset works).
2. LibreOffice export to PDF (`cff_writer.pdf`).
3. Ghostscript round-trip with `-sDEVICE=pdfwrite -dEmbedAllFonts=true
   -dSubsetFonts=true` → `cff_type1c.pdf`.

The resulting PDF has:

- `/FontFile3` with `/Subtype /Type1C` (genuine CFF, confirmed via pikepdf).
- `/Encoding` dict: `/BaseEncoding /WinAnsiEncoding`, `/Differences
  [ 1 /T /h /e /space /q /u /i /c /k /b /r /o /w /n /f /x /zero /one /two
  /three /four /five /six /seven /eight /nine ]` — codes 1-26 mapped to the
  glyph names of the 31 source characters.
- `/ToUnicode` CMap (623 bytes): `26 beginbfrange <01><01><0054> <02><02><0068>
  ... <1a><1a><0039> endbfrange`. Correct: code 1→'T', code 26→'9'.
- `/FirstChar 1`, `/LastChar 26`.

Rendering this PDF through `SvgRenderer.RenderPage(doc, 0)`, the `<text>`
body comes out as the 31-character string:

```
char codes: 32,32,32,32,32,32,32,32,9,32,10,32,32,13,32,32,32,32,32,32,32,32,32,32,32,32,32,32,32,32,32
```

The three values that leak through verbatim (9, 10, 13) are the XML-legal
control characters; all other low control codes (1-8, 11, 12, 14-31) get
replaced with space (32) by the XML writer. The 31 positions correspond
exactly to "The quick brown fox 0123456789", with the leak positions
matching the original codes 9 ('k'), 10 ('b'), 13 ('w').

## Root cause analysis

`Chuvadi.Pdf.Rendering.DisplayList.DisplayListBuilder.Builder.DecodeText`
(in `src/Chuvadi.Pdf.Rendering.DisplayList/DisplayListBuilder.cs`, around
line 840):

```csharp
private string DecodeText(byte[] bytes, string fontKey)
{
    if (_resources is null) { return TryLatin(bytes); }
    if (!_resources.TryGetValue(PdfName.Intern("Font"), out PdfPrimitive? fonts))
    {
        return TryLatin(bytes);
    }
    PdfDictionary? fd = _doc.Objects.ResolveAs<PdfDictionary>(fonts);
    if (fd is null) { return TryLatin(bytes); }
    if (!fd.TryGetValue(PdfName.Intern(fontKey), out PdfPrimitive? fv)) { return TryLatin(bytes); }
    PdfDictionary? font = _doc.Objects.ResolveAs<PdfDictionary>(fv);
    if (font is null) { return TryLatin(bytes); }
    try
    {
        PdfFont pf = PdfFont.FromDictionary(font, _doc.Objects);
        return pf.Decode(bytes);
    }
    catch { return TryLatin(bytes); }
}

private static string TryLatin(byte[] bytes)
    => System.Text.Encoding.Latin1.GetString(bytes);
```

`TryLatin` returns the raw byte as a 1-char string (`Encoding.Latin1` is
identity for 0-255). For byte 0x01 this is `\u0001`; for 0x09, `\u0009`;
for 0x54 ('T'), '\u0054'. This is the exact source of the observed output.

Six early-return paths funnel to `TryLatin` (five guard checks + the
swallowed `catch`). Without a diagnostic probe we don't know which fires
for `cff_type1c.pdf`. `PdfFont` and `CMapParser` are themselves correct:
running `pf.Decode` directly on the same font dict against bytes [0x01]
should return "T". So the bug is somewhere in the path that fails to reach
line 855.

### Candidate causes (need to be verified before fixing)

**Candidate 1 — `PdfDictionary` reference-equality vs `PdfName.Intern`.**
`PdfDictionary` (uploaded `1779981188319_PdfDictionary.cs`) declares its
internal store with `ReferenceEqualityComparer.Instance`:

```csharp
_entries = new Dictionary<PdfName, PdfPrimitive>(ReferenceEqualityComparer.Instance);
```

Lookups via `TryGetValue(PdfName.Intern("Font"), ...)` only succeed if the
exact same `PdfName` instance was used as the key during dict population.
If the PDF parser populated `_resources` with a `PdfName` instance that is
not the canonical interned one, the lookup silently fails. Same risk for
the inner `fd.TryGetValue(PdfName.Intern(fontKey), ...)` lookup. This is a
plausible cause if Ghostscript's PDF uses a Name that the parser doesn't
route through `PdfName.Intern`.

**Candidate 2 — inherited page Resources.** PDF 32000-1 §7.7.3.4 allows
Resources to be inherited from a parent page-tree node. If `PdfPage.Resources`
doesn't walk parents, a page that relies on inherited Resources gets
`_resources = null` (first guard fires). Less likely for `cff_type1c.pdf`
(a single-page Ghostscript PDF usually has direct Resources), but worth
ruling out.

**Candidate 3 — `ResolveAs<PdfDictionary>` returning null.** If the `/Font`
sub-dict or the per-key font dict is reached via an indirect reference
that resolves to something other than `PdfDictionary` (or to null), the
guard at line 847/850 fires. Should not happen for a well-formed PDF but
worth observing.

**Candidate 4 — silent exception in `PdfFont.FromDictionary`.** The `catch`
at line 857 swallows everything. If `PdfFont.FromDictionary` throws on
this font dict for any reason (e.g., a malformed encoding entry or
ToUnicode stream), the fallback hides it.

## Suggested next diagnostic (before any fix)

Extend the throwaway probe to instrument `DecodeText`. Minimum addition:
log which guard fires for this font dict on this PDF. A reasonable
approach is to add temporary diagnostic instrumentation to `DecodeText`
(perhaps gated by a static flag) that records which line returned, and
make the `catch` re-throw or at least log. Then re-run the probe against
`cff_type1c.pdf` to localize.

Alternative without modifying `DisplayListBuilder`: have the probe build
a `PdfFont` directly from the page's `/Resources/Font/<key>` entry using
the same lookup logic as `DecodeText`, and report at each step whether
the lookup succeeded. If every step succeeds when done from the probe but
fails inside `DecodeText`, the difference is the call site / state — likely
related to *when* `_resources` is set vs read.

## Suggested fixes (after the diagnostic localizes the cause)

### If Candidate 1 (reference-equality):

- Either change `PdfDictionary._entries` to use `EqualityComparer<PdfName>.Default`
  (which would require `PdfName` to implement `Equals`/`GetHashCode` by value),
  or audit every site that populates a `PdfDictionary` to ensure it always
  routes Names through `PdfName.Intern`. The former is safer (defends against
  future regressions of the same kind) but is a wider change.
- Tests: add a regression test that constructs a `PdfDictionary` with a raw
  `new PdfName("Foo")` and asserts `TryGetValue(PdfName.Intern("Foo"))` succeeds.

### If Candidate 2 (inheritance):

- Make `PdfPage.Resources` walk parents per §7.7.3.4 until a `/Resources`
  entry is found or the page-tree root is reached.
- Tests: synthesize a multi-page PDF where only the parent has `/Resources`,
  assert text on a child page decodes correctly.

### If Candidate 3 or 4 (silent failures):

- Replace the `TryLatin` fallback with logging-on-fallback at minimum. Better:
  surface a typed warning/diagnostic on the `PageDisplayList` so callers can
  detect when decode degraded. The silent fallback is the *real* defect — even
  if the lookup chain fails for legitimate reasons, swallowing it produces
  garbage output that looks fine in the SVG structure.

### Architectural cleanup (regardless of root cause)

`DecodeText` currently constructs a fresh `PdfFont` on **every single
character decode** (line 854) — a 31-character string causes 31
`PdfFont.FromDictionary` calls, each re-parsing the ToUnicode CMap. Not a
correctness bug, but a performance pothole and a smell that suggests the
font lookup belongs at the call sites that set `FontKey`, with the
resolved `PdfFont` cached per-key.

## Regression test ideas

- A test PDF (synthetic or `cff_type1c.pdf` if committable) whose decoded
  text contains a known non-trivial string, asserted character-for-character.
- A test that constructs a `BuilderState` with `FontKey` set but
  `_resources` arranged in each of the inheritance scenarios, asserts
  decode succeeds.
- A "no silent fallback" test: when `DecodeText` cannot resolve a font,
  the resulting display list carries a typed warning or the resulting glyph
  carries a sentinel that downstream renderers can detect.

## Relationship to v2.1.6

v2.1.6 (CFF embedding via `FontEmbedder`) is unaffected at the unit level:
the `OpenTypeFontBuilder`, `CffLoader.BuildCharsetMap`, and
`FontEmbedder.TryWrapCff` logic are all correct as written and pass their
unit tests. What v2.1.6 cannot prove until v2.1.7 lands is the **visible
end-to-end** rendering of a Type1C PDF in a browser: the font wrapping is
correct, but the text content emitted alongside it is currently broken by
this pre-existing decode fallback. Once v2.1.7 fixes decode, re-running
the SVG probe on `cff_type1c.pdf` should yield a correctly-rendered page
with the wrapped CFF font.
