# v2.1.8 and beyond — follow-up items

Originally written during v2.1.7 close-out to capture items discovered
that did not block v2.1.7. Updated during v2.1.8 close-out with what was
shipped, what remains, and what new gaps were found along the way.

## v2.1.8 status (this release)

**Shipped:**

- **Item 1** — chained `/Filter` array handling: `ObjectStreamReader.Decode`
  promoted from `private static` to `internal static` and now serves as the
  single implementation. `PdfReader.DecodeStreamBytes` delegates to it.
  Both single-Name and Array forms of `/Filter` are honoured. Five
  regression tests in `ChainedFilterDecodeTests` exercise the array,
  single-Name, no-filter, single-name-in-array, and malformed (non-Name
  entry) cases.
- **Item 3** — typed `RenderingDiagnostic` surface: new public record
  `RenderingDiagnostic(DiagnosticKind, string)` in
  `Chuvadi.Pdf.Rendering.DisplayList`. New `PageDisplayList.Diagnostics`
  property surfaces graceful-degradation events accumulated during build.
  `DisplayListBuilder.DecodeText` now appends a diagnostic at each of the
  six Latin-1 fallback paths, deduplicated by `(Kind, Message)` so a
  per-character condition emits one diagnostic per page, not one per glyph.
  Four constructor invariant tests in a new
  `Chuvadi.Pdf.Rendering.DisplayList.Tests` project.

**Deferred (intentional):**

- **Item 2** — `DecodeText` per-character `PdfFont` rebuild perf refactor.
  Pure refactor, no correctness change. Deferred to v2.1.9+ to keep v2.1.8
  scoped to the two real bug-fix items.
- **Item 4** — 12 stale local branches cleanup. Operational, do whenever
  convenient.
- **Item 5** — `page3-decodes.txt` / `page3-stream.txt` debug-dump
  removal from repo root. Operational.
- **Item 6** — CFF fixture mystery confirmation (almost certainly
  fixture-specific based on v2.1.7's real-world rendering evidence;
  worth one confirmation run when next touching CFF).

**Attempted and rolled back:**

- **Item 7** — docs/api/* line-ending churn on regen. v2.1.8 added
  `newline="\n"` to the two `write_text` calls in `tools/gen_api_docs.py`.
  After running the regen, `git status` reported all 312 docs/api/*.md
  files as modified. Per-file `git diff` returned empty output on the
  files spot-checked, meaning no visible content difference. `git
  update-index --refresh` reported every file (including hand-edited
  source files unrelated to docs) as "needs update" — i.e. the refresh
  state was telling us about cached stat, not content drift, and on
  closer inspection my reading of `needs update` was less load-bearing
  than I initially thought.

  The change was reverted: `tools/gen_api_docs.py` was restored to its
  v2.1.7 state. The symptom we set out to fix is still present; the
  hypothesis we tried (CRLF → LF translation) was not confirmed by data
  and likely is not the root cause. Item 7 is **re-opened with a
  warning**: do not try a similar small fix until the mechanism is
  actually diagnosed. Specifically:
  - Determine *what* exact bytes differ between regen output and HEAD,
    via something like `git diff --binary docs/api/<one-file>.md | head`
    or by comparing file hashes before/after regen.
  - If only line endings, confirm the *direction* of mismatch (repo LF
    vs working CRLF, or repo CRLF vs working LF) — a working-copy byte
    scan and an `git cat-file -p HEAD:<file>` byte scan, compared.
  - Once cause is known, fix it confidently. Until then, the regen
    inconveniently produces "modified" status on docs each run; deal
    with by `git checkout -- docs/api/` after each regen (the
    workaround we used in v2.1.7 close-out).

## Discovered during v2.1.8, new items

- **Item 8 (new)** — `Chuvadi.slnx` is missing the
  `Chuvadi.Pdf.Rendering.DisplayList` src project. The project builds
  fine because `dotnet build` discovers `.csproj` files independently
  of the solution file, but IDEs that respect `.slnx` (Visual Studio,
  JetBrains Rider) would not show this project. v2.1.8 added the new
  `Chuvadi.Pdf.Rendering.DisplayList.Tests` project to the `.slnx`,
  but deliberately did not add the missing src project — the audit
  belongs in its own change. Audit when convenient: open `Chuvadi.slnx`,
  cross-reference against every `*.csproj` under `src/`, `tests/`, and
  `examples/`, and add any that are missing.

- **Item 9 (new)** — `tools/gen_api_docs.py` does not emit a doc page
  for the public record `RenderingDiagnostic` (added in v2.1.8 item 3).
  The companion enum `DiagnosticKind` got a page; the record did not.
  Best guess: the generator does not yet handle C# records. Worth a
  small fix to the generator. Until then, the public surface is
  documented in XML doc comments on `RenderingDiagnostic.cs` itself.

## Notes on chained `/DecodeParms`

Per PDF 32000-1:2008 §7.4, `/DecodeParms` is `/Filter`'s sibling:
per-filter parameters. The current `Decode` (shared between
`PdfReader.DecodeStreamBytes` and `ObjectStreamReader`) ignores
`/DecodeParms` entirely. This was already the case in v2.1.7 and earlier.
Worth fixing when a real-world PDF surfaces that needs them. The fix is
non-trivial — `FilterPipeline.Decode` already accepts a `FilterParameters`
argument, but threading the right parms per filter requires parsing the
array form as well, which doubles complexity.
