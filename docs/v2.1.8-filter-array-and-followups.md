# v2.1.8 (or later) — Follow-up items from v2.1.7

This document records items identified during v2.1.7 development that
are not blocking the v2.1.7 release but should be addressed in a future
phase.

## 1. Single-Name vs Array `/Filter` handling

`PdfReader.DecodeStreamBytes` currently handles only the single-Name
`/Filter` case:

```csharp
private static byte[] DecodeStreamBytes(PdfDictionary dict, byte[] rawBytes)
{
    PdfName? filterName = dict.GetName(PdfName.Filter);
    if (filterName is null) { return rawBytes; }
    FilterPipeline pipeline = FilterRegistry.CreateDefaultPipeline();
    string resolvedFilter = FilterRegistry.ResolveAlias(filterName.Value);
    return pipeline.Decode(resolvedFilter, rawBytes);
}
```

Per PDF 32000-1:2008 §7.4, `/Filter` can also be an **Array** of Names
naming a chain of filters that were applied in order (e.g.
`/Filter [/ASCII85Decode /FlateDecode]`). When `/Filter` is an array,
`GetName` returns null and `DecodeStreamBytes` silently returns the raw
(undecoded) bytes — which is wrong.

`ObjectStreamReader.Decode` (added in v2.1.7) does handle both cases; the
implementation is a useful starting reference. The fix is to lift that
implementation into `PdfReader.DecodeStreamBytes` (or share it via a
common helper). When you do, also lift `/DecodeParms` handling — the
array form often has per-filter parameters that today's code ignores.

**Why deferred:** No bug reproducer in the current test corpus. Object
streams that we now resolve are almost always single-filter (FlateDecode).
Real-world content streams with chained filters are rare. Fix it before
the first end-user complaint or as part of a "filter pipeline hardening"
phase.

**Risk if not fixed:** PDFs authored with chained filters (e.g. some
older Acrobat output, some PostScript-to-PDF converters) silently mis-
decode their streams. Failure mode is "the stream renders as garbage" —
not a security issue, just wrong output.

## 2. Architectural cleanup in `DisplayListBuilder.DecodeText`

(Carried over from the v2.1.7 ticket.) `DecodeText` constructs a fresh
`PdfFont` on **every single character decode** — for a 31-char string,
31 `PdfFont.FromDictionary` calls each re-parsing the ToUnicode CMap.
Not a correctness bug, but a meaningful perf pothole.

The right fix is to cache the resolved `PdfFont` per `FontKey` once,
ideally at the point where `FontKey` is set (the `Tf` operator handler),
and just reuse that cached `PdfFont` in every `DecodeText` call until
`FontKey` changes again.

**Why deferred:** Pure refactor, no correctness change. Want to ship
v2.1.7 with the critical bug fixed first; the perf work can be a focused
follow-up.

## 3. `DecodeText` silent-fallback hygiene

Even after v2.1.7 fixes the underlying object-resolution bug, the
`TryLatin` (Latin-1 byte-passthrough) fallback in `DecodeText` is still
a silent way for new bugs of the same shape to hide.

Recommended:

- Replace the silent fallback with a logged-on-fallback (debug log or a
  warning sink). The Latin-1 passthrough is sometimes a legitimate
  recovery for genuinely malformed PDFs and we don't want to throw, but
  it should leave an audit trail.
- Alternatively, surface a typed warning on the `PageDisplayList` so
  callers can detect decode degradation.

## 6. CFF fixture mystery from v2.1.6

(Carried over from v2.1.6.) When validating CFF embedding in v2.1.6, a
fontTools-built CFF passed as a base64 fixture to `new CffLoader(...)`
yielded an empty `GlyphNameToGid`. Cause unresolved; we shipped without
the test that depended on it.

Once v2.1.7 unblocks real Type1C PDF rendering, run a real CFF through
`CffLoader` and confirm `GlyphNameToGid` is populated. If it is, the
v2.1.6 mystery was fixture-specific (malformed bytes or a fontTools
quirk we don't support); if it's not, `CffLoader` has a real parser
limitation worth fixing.

**Update from v2.1.7:** rendering of `cff_type1c.pdf` end-to-end through
the SVG pipeline now produces a correctly-embedded font with all 26
characters of "The quick brown fox 0123456789" visible in the browser.
This is strong evidence that `CffLoader.GlyphNameToGid` is populated
correctly on real Ghostscript-produced Type1C charsets. The v2.1.6
fixture mystery is most likely fixture-specific (the fontTools-built CFF
base64 we embedded). Worth confirming next time we touch CFF: build a
new minimal CFF with fontTools, feed it through `CffLoader`, observe
`GlyphNameToGid`. If empty, dig in; if populated, close this item as
fixture-specific.

## 7. docs/api/* line-ending churn on every regeneration

`python tools\gen_api_docs.py` on Windows writes `docs/api/*.md` files
with CRLF line endings. The repo's stored copies are LF. Every regen
therefore produces a diff (in the working copy) that is content-empty —
git's `core.autocrlf` reports "CRLF will be replaced by LF" warnings on
each touched file. Saw this on the v2.1.6 merge and again on v2.1.7
close-out.

**Cost:** every release contributor has to either revert no-op diffs
before committing, or accept a noisy PR. The noise hides any real
content changes from the regen.

**Fix options:**

- Make the generator write LF directly (most robust). Open `docs/api/*.md`
  with `open(path, "w", newline="\n", encoding="utf-8")` or write bytes
  explicitly with `\n`.
- Pin `*.md` (or just `docs/api/*.md`) to LF via `.gitattributes`. Less
  robust because it only protects the committed bytes, not what's in the
  working copy — but avoids the warning fanfare.

The first is the durable fix.
