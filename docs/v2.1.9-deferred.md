# Deferred fixes after v2.1.9 — RESOLVED

Five items surfaced while completing v2.1.9 (item 8, "complete the slnx").
Each was a pre-existing defect that was invisible because the affected project
was excluded from `Chuvadi.slnx` and therefore never built or tested by CI.
Completing the solution exposed them.

All five are now fixed and verified (compile and runtime). This document records
the root causes and resolutions. The four affected test/example projects have
been added back to `Chuvadi.slnx` (64 -> 68 projects).

## Item 1 — Authoring.Tests: missing test package references — FIXED

`Chuvadi.Pdf.Authoring.Tests.csproj` declared only project references and no
PackageReference block, so xUnit and FluentAssertions were unresolved (CS0246).

Fix: added the standard four-package test block (xunit,
xunit.runner.visualstudio, Microsoft.NET.Test.Sdk, FluentAssertions) plus
AssemblyName/RootNamespace/NoWarn, modelled on Filters.Tests. Confirmed against
AuthoringTests.cs that those are the only external packages used.

## Item 2 — Cryptography.Tests: would not compile, then exposed 3 real failures — FIXED

The project had been excluded so long it had drifted out of sync with production.
Bringing it into the build surfaced two layers of problems.

### Layer 1 — 6 compile errors

- `RsaPssSignerTests.cs` — missing `using Chuvadi.Cryptography.Signing;`
  (RsaPssSigner exists; the test never imported its namespace).
- `TrustStoreTests.cs` — CS7036: `SubjectPublicKeyInfo` constructor gained a
  third `byte[] rawEncoding` parameter; the two-arg call no longer compiled.
  Fixed by passing a minimal placeholder (the tests assert on the Subject name
  only, never on spki.RawEncoding).
- `RsaSignerTests.cs` — IDE0051: removed the unused `CertDerHex` constant.
- `EcdsaPrivateKeyTests.cs` — IDE0005: removed unused `using ...Asn1;`.
- `EcdsaVerifierTests.cs` — IDE0005: removed unused `using System.Numerics;`
  (all references were fully qualified).
- `EcdsaCmsSignerTests.cs` — IDE0005: removed unused `using ...Oids;`.

### Layer 2 — 3 real test failures exposed once it compiled

- **#1 (test bug) `Asn1TagLengthTests.RoundTrip_Length_PreservesValue`**:
  the `[InlineData(0x7FFFFFFF)]` case allocated `header.Length + length` bytes,
  which overflows Int32 at int.MaxValue. Production was correct (it uses a
  `(long)` cast to guard the bounds check). Fix: kept the 4-octet round-trip
  coverage (0x01000000) and moved int.MaxValue into a new negative test,
  `Read_LengthExceedingBuffer_ThrowsAsn1Exception`, which verifies the
  over-large-length guard.
- **#2 (production bug) `SignatureVerifier.DispatchRsaPss`**:
  guarded `Parameters is not null`, but `AlgorithmIdentifier` normalises a null
  parameters argument to an empty array, so the guard was always true and the
  intended `NotSupportedException` branch was dead code. Paramless RSA-PSS
  therefore leaked an `Asn1Exception` from parsing empty bytes. Fix: guard
  `ParametersAreAbsent` (empty) instead of null, so paramless PSS correctly
  throws NotSupported.
- **#3 (production bug) `Asn1String.ReadUtf8`**:
  used `Encoding.UTF8.GetString`, which substitutes replacement characters and
  never throws, so the `catch (DecoderFallbackException)` was dead and invalid
  UTF-8 was silently accepted. Fix: decode with a strict
  `UTF8Encoding(throwOnInvalidBytes: true)` so malformed input is rejected as
  `Asn1Exception`. Only ReadUtf8 was changed; the ASCII/Latin1/BMP readers have
  their own validation and no failing test.

Result: 424/424 tests pass.

## Item 4 — Cli.Tests: inspect-xref returned exit 1 — FIXED

NOTE: the original tracking note described this as a "macos-only numeric .Be()
failure". That was wrong. The failure was `InspectXref_OnSamplePdf_PrintsHeader`
returning exit code 1 on ALL platforms, and the real cause was unrelated to
numerics or platform.

Root cause: `InspectXrefCommand` called `XrefTable.Parse(fs)` on a freshly
opened stream positioned at offset 0. `XrefTable.Parse` requires the stream to
be positioned at the xref table; at offset 0 it reads the `%PDF-...` header and
throws `Invalid xref subsection header`. The command never located `startxref`.
This failed on every platform and on every real PDF, not just the test fixture.
(The Info/DumpObjects commands passed because they use PdfDocument.Open, which
locates the xref internally.)

Fix (production): added a public `PdfReader.LoadXref(Stream)` that reuses the
existing `FindStartXref` + `LoadXrefChain` machinery (handles both classic xref
tables and xref streams, following /Prev links), and changed the command to call
it instead of `XrefTable.Parse`. Verified: inspect-xref now prints the xref
table and exits 0 on a real PDF; 38/38 Cli.Tests pass.

## Item 5 — gen_api_docs.py and check_style.py defects — FIXED

### 5a. gen_api_docs.py: record struct / readonly struct rendered without modifiers

A `readonly record struct` rendered as `record`, and `readonly struct` lost its
`readonly`. Cause: `kind = ...split()[0]` discarded the second word of
`record struct`, and `readonly` was never re-emitted. Fix: preserve the full
kind, add an `is_readonly` flag, key the label on the first word, and emit
`readonly`. Verified identical output on Python 3.12 and 3.14; corrected 22
pages.

### 5b. check_style.py: two false positives (Roslyn does not flag these)

- IDE0008 tuple deconstruction: the `var` regex matched `var (a, b)`. Fixed by
  exempting `var` immediately followed by `(`.
- CS0234 transitive references: `check_csproj` required a direct ProjectReference
  even when the namespace resolves transitively (e.g. Images -> Filters ->
  Primitives, confirmed via Filters.csproj). Rewrote to walk the transitive
  ProjectReference closure. A genuinely missing reference is still flagged.

### 5c. check_style.py: dead check_syntax block

`check_syntax` and its monkey-patch were defined after the `__main__` guard, so
they never executed (confirmed by git history: present since the file's creation
commit, never relocated). Removed.

Verified: zero false positives across all 336 files; real `var` misuse and real
missing references still flagged.
